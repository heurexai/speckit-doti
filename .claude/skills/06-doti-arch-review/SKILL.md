# 06-doti-arch-review

Read `.doti/agent-context.md`, then follow `.doti/core/templates/commands/doti-arch-review.md`.

A multi-lens DESIGN review before implementation. The stage ALWAYS runs (a hard prereq of /07) but its DEPTH is conditional (FR-028/SC-013): TRIAGE the change first, then activate ONLY the lenses it actually touches — a lens whose concern the change does not touch EXITS 'not applicable' and spawns no sub-agent. CRUCIAL triage split: 'templates' is not one thing — Doti PROSE (command templates under .doti/core/templates/commands/, skills, agent-context, docs) is no-code and skips the code lenses; but GENERATED-CODE templates under `scaffold/templates/**` (the .cs/.csproj/tests/rules that become a real generated repo) ARE code — run the code lenses there exactly like production ('it's a template' is no excuse). Run the applicable lenses (design-soundness, edge-case/failure-mode, data-contract, security, blast-radius, simpler-alternative, testability, fit-with-current-architecture) — each carries a pre-canned 'look for' checklist; if you are capable of sub-agents, run one clean-context sub-agent per lens IN PARALLEL — against the spec/plan/tasks + current code, with the changed-files context fetched ONCE (`hx impact plan --for arch-review --json`, or `git diff` fallback) and injected into every lens; each finding carries severity (BLOCKER/HIGH/MEDIUM/LOW) + evidence (`file:line` or spec/plan ref) + a fix. ALWAYS write the review to `docs/reviews/{NNN-slug}-arch-review.md` — the review record /08-drift-review reads (a future cycle binds it as the stage's produces artifact); a no-code change still writes a minimal no-op record naming the triage result, it does not skip the stage. Sentrux/ArchUnitNET are NOT run here — they measure implemented code and are enforced by `gate run` at `/07` (the `cliSurfaceConfinement`/`cliDelegation` ArchUnitNET families + the Sentrux layer/cycle boundaries). An open BLOCKER in an applicable lens, or a missing applicable lens, blocks `/07-doti-implement`; a correctly-skipped lens does not.

## Asking the operator a question (required format)

Any operator-facing question — at any stage, for Claude or Codex — uses this fixed format, presented immediately before the question:

- **Context** — the full background. Assume the operator has not re-read the material; restate what is unclear and the relevant facts.
- **Why it matters** — the concrete impact and why it needs a decision now.
- **Options** — for each option: **Pros**, **Cons**, and **Consequence** (what becomes true downstream if chosen).
- **Recommendation** — the option you recommend and the reasoning; list it first and label it "(Recommended)".
- **Assumptions** — the assumptions behind the recommendation.
- **Confidence** — High / Medium / Low, with a one-line reason.

Evidence requirement: every question, option, recommendation, and assumption must rest on verified facts. Verify each premise first (read the code, run the tool, observe the output) and cite that evidence; never present options whose premises you have not confirmed. An assumption is allowed only when the fact cannot be obtained from this environment — then label it UNVERIFIED, state why it cannot be verified and what would verify it. Prove claims (reproduce/RCA); never assert them. Do not use "assume" to defer work you could do — an unproven premise is a defect that can ruin the design.
