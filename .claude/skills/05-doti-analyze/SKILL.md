## Asking the operator a question (required format)

Any operator-facing question â€” at any stage, for Claude or Codex â€” uses this fixed format, presented immediately before the question:

- **Context** â€” the full background. Assume the operator has not re-read the material; restate what is unclear and the relevant facts.
- **Why it matters** â€” the concrete impact AND a brief **concrete example** of how it plays out, and why it needs a decision now. Frame the stakes as QUALITY: for a code change â€” maintainability, consistency with existing patterns, correctness, testability, and future drift; for a docs/Doti-prose change â€” clarity, consistency, and accuracy of the documentation. **Never frame the stakes as effort or convenience** â€” that one option is more work is not why a question matters.
- **Options** â€” for each option: **Pros**, **Cons**, and **Consequence** â€” what becomes true downstream if chosen, reasoned through in those same quality terms with a **concrete example** (e.g. "then every new command must re-declare its error codes â€” a consistency cost paid forever", or "the README would then document two install paths â€” a reader-clarity cost"). Effort/convenience is NOT a pro, con, or consequence here; whether an option is more work never decides it.
- **Recommendation** â€” the option you recommend and the reasoning â€” the quality trade-off you are optimising, never "the easy one"; list it first and label it "(Recommended)".
- **Assumptions** â€” the assumptions behind the recommendation.
- **Confidence** â€” High / Medium / Low, with a one-line reason.

Evidence requirement: every question, option, recommendation, and assumption must rest on verified facts. Verify each premise first (read the code, run the tool, observe the output) and cite that evidence; never present options whose premises you have not confirmed. An assumption is allowed only when the fact cannot be obtained from this environment â€” then label it UNVERIFIED, state why it cannot be verified and what would verify it. Prove claims (reproduce/RCA); never assert them. Do not use "assume" to defer work you could do â€” an unproven premise is a defect that can ruin the design.
