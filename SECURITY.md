# Security Policy

## Supported versions

Security fixes are handled on `main`. Until the project publishes versioned releases, treat the latest commit on `main` as the only supported line.

## Reporting a vulnerability

Please do not report security vulnerabilities in a public issue.

Use GitHub private vulnerability reporting for this repository when it is enabled. If private vulnerability reporting is not available yet, open a minimal public issue asking for a private security contact path and do not include exploit details, logs, proof-of-concept code, secrets, or affected private systems.

Helpful private reports include:

- affected component or command;
- exact version, commit, or branch;
- impact and prerequisites;
- minimal reproduction steps;
- whether the issue affects generated projects, this scaffold repository, or both.

## What to expect

Maintainers will triage privately, confirm the affected scope, and coordinate a fix before public disclosure. The repository's build, analyzer, package-audit, Gitleaks, and deterministic gate checks are part of the validation path for security fixes.
