# 029 — Implementation plan: scaffold & doti setup configuration

> Revised after `/04-doti-arch-review` (2 BLOCKERs + 7 HIGHs). Resolutions are folded into the decisions below and mapped in `docs/reviews/029-scaffold-setup-config-arch-review.md`. The decisive change: the **mutating** units (projector orchestration, writers, store, formatter) are re-homed so no `Doti.Core→Scaffold.Core` edge (a Sentrux `max_cycles=0` violation) or `Runner.Cli→Scaffold.Core` edge (composition inversion) is ever required.

## Summary

Put the **pure** config model, resolver, schema, defaults, prompt-definitions, AND the projection **orchestration** in `Hx.Tooling.Contracts` (which both surfaces already depend on), with each projection target modeled as an **injected writer delegate**. Home the concrete **`.doti`-asset writers** (release.json, GitVersion.yml, `.csproj` metadata, constitution §2, the setup-config store + formatter) in `Hx.Doti.Core` beside `RepoPayloadStore`/`ConstitutionInitializer` — reachable by `hx new` (Scaffold.Core→Doti.Core), `hx doti install` (Doti.Core), and `config show` (Runner.Cli→Doti.Core) with **zero new edges**. Validate everything in the CLI **before** generation so SC-006 holds; encode/contain all projected values; persist only repo-portable intent to a tracked `.doti/setup.json`; render with provenance via `hx doti config show`. CLI stays parse→delegate→render.

## Existing-architecture assessment

- **Edges (verified):** `Hx.Scaffold.Core → Hx.Doti.Core` exists (`src/Hx.Scaffold.Core/Hx.Scaffold.Core.csproj`). `Hx.Runner.Cli → Hx.Doti.Core` exists; `Hx.Runner.Cli` has **no** `Hx.Scaffold.Core` edge. `Hx.Doti.Core → Hx.Tooling.Contracts` and `Hx.Scaffold.Core → Hx.Tooling.Contracts` exist. ⇒ a `Doti.Core→Scaffold.Core` call **cycles** (Sentrux `max_cycles=0`); Contracts is reachable by all; Doti.Core is reachable by all three call sites.
- **Scaffold pipeline** — `ScaffoldNewRunner.Run(ScaffoldRequest, …)`: `TemplateGenerator.Generate` writes the full tree directly into `OutputPath` (no sandbox/rollback) → `ToolVendor` → `ScaffoldDotiInstaller.Install` → `FirstSmokeRunner.Run` (sets the Sentrux baseline) → `ScaffoldHookArmorer.Arm`. **The finish phases already throw on failure, leaving a partial repo** (the existing failure model).
- **`.doti`-asset owners (Doti.Core):** `RepoPayloadStore`/`ManagedAssetManifestStore` (managed assets), `ConstitutionInitializer` (write-once `{PROJECT_NAME}` token only — no §2 fill), `ConstitutionService` (`## §2` anchor + `ExtractSection2`). Release manifest writer `ReleaseTargetManifest`/`ScaffoldReleaseTargetWriter` currently in `Hx.Scaffold.Core/Release` → **to be re-homed/duplicated into Doti.Core** (see D3).
- **CLI bind** — `ScaffoldCommandFactory.AddNew` → `ScaffoldCommands.New(...)` (wrapped in `WithRequiredConfiguration`); `RunnerCommandFactory.Doti.AddDotiInstall` → `RunnerCommands.DotiInstall` → `DotiInstallBootstrapper.Install(DotiInstallBootstrapRequest)`. The install request already carries additive-trailing-optional fields (`RepositoryName = null, Force = false`).
- **Guards/discipline:** `schemaVersion == 1` fail-closed (release.json/hx.config.json); `ReleaseTargetManifest.ValidatePublishProject` (no `..`, `GetFullPath`+containment) is the path-containment pattern; ArchUnit `cliSurfaceConfinement` forbids `.Cli` types suffixed Service/Repository/Validator/Scanner/Runner/Resolver/Engine/Provider/Planner/Classifier/Projector/Detector; `cliDelegation` requires `*Commands` to depend on a `*.Core` namespace.

## Design decisions

**D1 — Pure model + resolver + schema + defaults + prompt-definitions in `Hx.Tooling.Contracts`.** `SetupConfig` (deserialized DTO, all-nullable), `ResolvedSetupConfig` (per-key `ConfigField{Value, Source, Default}` + a `SetupKey` descriptor carrying **`Group`** (identity|versioning|release|publish|agents|constitution) and **`AppliesTo`** (New|Install|Both)), `ConfigSource` enum, `SetupConfigDefaults` (the single source of defaults, asserted == `docs/configuration.md`), `SetupConfigResolver.Resolve(raw, flagOverrides, audience)` (pure), `SetupConfigSchema.Validate` (pure), `SetupPromptDefinitions` (pure data incl. an `EnabledWhen` for conditional branches). Verified Contracts is IO-free (System.Text.Json only) — keep it so. *(Resolves the Contracts-purity check; LOW group-drift via the `Group` tag.)*

**D2 — Provenance first-class, resolved in two stages.** Layer inputs (flag ▸ config-file ▸ interactive ▸ derived ▸ default) recording the winning `Source` per key → `ResolvedSetupConfig`; then project. *Unchanged.*

**D3 — Projection ORCHESTRATION in Contracts; WRITERS injected, homed at their asset. [resolves BLOCKER-1, HIGH fit, HIGH simpler, MEDIUM §2-ownership]** `SetupConfigProjector.Project(resolved, audience, writers)` lives in **Contracts** and iterates the single key→target table (each row tags `AppliesTo`); each target is an **injected `ISetupTargetWriter`** the caller supplies. Concrete writers live where their asset lives: **template-token / project-generation writers → `Hx.Scaffold.Core`** (New-only); **`.doti`-asset writers → `Hx.Doti.Core`** — a `ReleaseTargetWriter` (re-homed/duplicated from Scaffold.Core), a `GitVersionSeedWriter`, a `CsprojMetadataWriter`, and a `ConstitutionSection2Writer` (new, beside `ConstitutionInitializer`, reusing `ConstitutionService.Section2Anchor`, write-once-when-placeholder, idempotent). `hx new` supplies {Scaffold.Core + Doti.Core} writers via the existing Scaffold.Core→Doti.Core edge; `hx doti install` supplies the Doti.Core Install-subset writers directly; **no `Doti.Core→Scaffold.Core` edge, no cycle.** New-only keys reached on the install path are collected and **reported as ignored** (not silently dropped).

**D4 — Wizard: thin CLI IO loop over pure prompt-definitions. [resolves MEDIUM wizard-sprawl, LOW naming]** The CLI type is a dumb iterator (neutral name — `SetupWizard`/`SetupPrompts`/`SetupWizardLoop`, none a forbidden `cliSurfaceConfinement` suffix) that reads `SetupPromptDefinitions` (Contracts), echoes each definition's "what it affects", collects answers into a `SetupConfig`, then re-enters the `--config` resolve+project path. Conditional branches (`publish=yes → NuGet sub-questions`) are **data** (`EnabledWhen` on the definition), not CLI branching. IO is behind an injectable `IConsole`/reader for testability (D-test). Final **Review** step renders `config show` (human).

**D5 — Validate in the CLI BEFORE generation; bound the atomicity contract. [resolves BLOCKER-2, HIGH validate-order, MEDIUM SemVer/SPDX]** `SetupConfigSchema.Validate` + the `--config` read + value checks (`schemaVersion==1`, unknown-field, `agents ⊆ {claude,codex}`, **`nextVersion` = 3-part numeric SemVer core via one shared parser**, **`license` = SPDX id/expression charset**, free-text XML-metachar/control-char rejection, **path containment** of `identity.output`/`--config` path) run in `ScaffoldCommands.New` / the install command **before** the request record is built and therefore before `ScaffoldNewRunner.Run`/`TemplateGenerator`. ⇒ an invalid config → **no generation → no partial repo (SC-006)**. The atomicity guarantee is **scoped to validation**; a failure during the post-generation projection follows the *existing* scaffold finish-phase model (throw, like `ToolVendor`/`ScaffoldDotiInstaller` already do) — SC-006 does not promise rollback of a non-validation failure, and the spec's SC-006 (invalid value) is satisfied as written.

**D6 — Persist repo-portable intent only; idempotent; report preserved. [resolves MEDIUM machine-path leak, LOW §2-ignored]** The operator-supplied (non-default) **repo-portable** fields are written to a tracked `.doti/setup.json`; **machine-local fields (`localOutput.directory` and any absolute path) are NEVER written to the tracked file** — they route to the executable-adjacent `hx.config.json` template (or are rejected from the tracked file). Re-runs are idempotent (matching fields no-op; operator edits preserved via the existing managed-asset reconciliation). A config that carries `constitution.*` while §2 is already authored emits a **preserved/ignored effect** (mirroring `DotiInstaller`'s existing preserved-effect emission) so nothing is silently dropped.

**D7 — `config show` (Runner.Cli) reads via a Doti.Core store; persisted-only; all-default when absent. [resolves MEDIUM config-show placement, MEDIUM no-file]** `hx doti config show [--json]` lives in `Hx.Runner.Cli`; the `SetupConfigStore` (read/write `.doti/setup.json`) and the human-table formatter are homed in **`Hx.Doti.Core`** (beside `RepoPayloadStore` — `.doti/setup.json` is a Doti asset like `.doti/payload.json`), reachable by the existing Runner.Cli→Doti.Core edge; the resolver is in Contracts. When **no `.doti/setup.json` is present, render the all-default view** (every key `Source==Default`), not an error. Renders: JSON (`{value, source, default}`) or a human table grouped by `Group` with a `N custom · M default` footer. Non-mutating.

**D8 — Both request records carry `Setup`; install has its own seam. [resolves HIGH install-seam, MEDIUM D8-install, LOW ScaffoldProof]** Add an additive optional `ResolvedSetupConfig? Setup = null` to **both** `ScaffoldRequest` (hx new) **and** `DotiInstallBootstrapRequest` (hx doti install). `ScaffoldNewRunner` runs the full projection after `TemplateGenerator`; `DotiInstaller` runs the Install-subset projection. `ScaffoldProof.Request.Setup` is an additive JSON field (null/omitted on the no-config path → SC-007 JSON shape preserved); `ResolvedSetupConfig` serializes cleanly.

**D9 — Security guards at the projection seam. [resolves HIGH csproj-XML, HIGH path-traversal, LOW §2-injection]** Every value written into `.csproj`/`.props`/`GitVersion.yml` is **encoded** (`System.Security.SecurityElement.Escape` or `XElement`/`XDocument` so the framework escapes) and free-text fields reject control chars + XML metacharacters at validation (D5). `identity.output` + the `--config` path are **contained** (reuse the `ValidatePublishProject` discipline: reject `..`, `GetFullPath`+`StartsWith` the intended base). The projected §2 prose is checked for **anchor integrity** (reject a forged `## §1`/`## §2` heading that would break `ConstitutionService.ExtractSection2`); the spec documents the trust assumption (the `--config` author is the operator).

**D10 — Provable no-op fence when `Setup` is absent. [resolves HIGH/MEDIUM SC-007 both surfaces]** When `Setup` is null/absent (no `--config`, no `.doti/setup.json`), `SetupConfigProjector.Project` **early-returns before any write** — for both `hx new` (SC-007 byte-identical) and `hx doti install` (the install no-config path stays byte-identical; `doti update`/`update-all` unaffected). A regression test pins each.

## Module breakdown (re-homed)

| `ReleaseTargetWriter`, `GitVersionSeedWriter`, `CsprojMetadataWriter`, `ConstitutionSection2Writer`, `SetupConfigStore`, `SetupConfigTableFormatter` | `Hx.Doti.Core` | `.doti`-asset writers beside `RepoPayloadStore`/`ConstitutionInitializer`; reachable by new+install+show, no cycle |

## Architecture delta

No new project; **no new project edge; no cycle** (the decisive fix). Orchestration + model pure in Contracts; `.doti`-asset writers in Doti.Core; project writers in Scaffold.Core; CLI thin (wizard named neutrally, delegates — `cliSurfaceConfinement`/`cliDelegation` hold). `ScaffoldRequest`/`DotiInstallBootstrapRequest`/`ScaffoldProof` gain additive optional fields. No gate downgrade; validation fail-closed; Sentrux function-size respected (writers are small single-purpose units).

## Test strategy (incl. the testability lens)

- **Pure unit:** resolver provenance per audience; schema fail-closed (schemaVersion, enum, **SemVer**, **SPDX**, XML-metachar, **path-traversal**) each naming the field; the projection table via **injected fake `ISetupTargetWriter`s** (each key→target, audience-scoped, New-only-ignored-on-install); prompt-definitions ↔ key 1:1; **the D10 no-op fence (Setup null → zero writer calls)**.
- **Wizard IO seam:** an injectable `IConsole`/reader so the loop is driven by **scripted input** — SC-004 (`--interactive` == `--config`) testable with no TTY.
- **Integration:** SC-001..SC-008; `hx new --config` with `description="A & B"` → a **valid** `.csproj` (D9 encode); `--output ../escape` → rejected (D9 containment); `config show` on a no-config repo → all-default (D7); **`hx doti install` no-config → byte-identical** (D10); `.doti/setup.json` never contains an absolute `localOutput.directory` (D6).
- **Golden:** a `doti-setup.json` fixture + expected `config show --json`; `SetupConfigDefaults` == `docs/configuration.md` assertion.

## Risks

- **Writer re-homing churn** (ReleaseTargetWriter moves to Doti.Core) — mitigate by duplicating the small writer and pointing `ScaffoldReleaseTargetWriter` at it (or moving + re-referencing via the existing Scaffold.Core→Doti.Core edge); a test asserts `hx new`'s release.json is unchanged.
- **Wizard IO** — covered by the `IConsole` seam.
- **Request-record ripple** — additive optional fields; existing callers compile (SC-007 locks both surfaces).
- **Default drift** — `SetupConfigDefaults` single source + the docs-parity assertion.
