## BLOCKERS (must be closed before /07 — all resolved in-cycle)

### BL-1 — Sentrux band/two-try is wired to the wrong arm; the band's signal is unspecified
**Lens:** A. **Evidence:** `SentruxChecker.Regression.cs:13-21` — `SentruxRegression.Evaluate` runs first, then `gate.Degraded` **overrides** the outcome to `Fail` (`:18-21`), and `SignalBefore is null` overrides to `Blocked`. Plan 2e + T024/T025 extend only `SentruxRegression.Evaluate` (`SentruxRegression.cs:12`) and `Hx.Cycle.Core` — they never name `SentruxChecker.Regression.cs`, the wiring file that consumes `Evaluate`, applies the `Degraded` override, and builds the `RegressionCheck` the gate proof carries. As written, a three-state `Evaluate` result cannot reach the cycle's two-try log, and it is undefined whether the "130 band" measures the quality-signal *deviation* or the structural `Degraded` flag.
**Why BLOCKER:** the operator's design (band = complexity **deviation**, two tries, then structural arch-review; `Degraded` = wrong-architecture **hard stop**) is unimplementable through the un-named wiring; and conflating `Degraded` into the band would wrongly grant a genuine structural-rule violation two tries + a raised limit.
**Resolution (plan 2e, T024/T025/T026 amended):** (a) the band measures the **quality-signal deviation** (`Evaluate` arithmetic) — the operator's complexity-deviation; (b) `gate.Degraded` (absolute structural-rule violation) stays an **immediate hard `Fail`**, never band-eligible; (c) name `SentruxChecker.Regression.cs` as the wiring point and propagate the three-state outcome through `RegressionCheck` → `GateProof` so the `Hx.Cycle.Core` two-try log reads `EscalationBand`; (d) the "two attempts → structural arch-review" applies to `EscalationBand` only. Grounded in the operator's stated band semantics.

### BL-2 — FR-038 fix is unsatisfiable: `CommitScope` is boolean, not a path-set (`CommitScopeInspector.cs` is a missed file)
**Lens:** E + B (converged). **Evidence:** `CommitScopeInspector.cs:7-13` — `CommitScope` exposes `HasUnstagedTrackedChanges`/`HasUntrackedChanges` as **bare booleans**; `ParseStatus:60-64` sets the untracked flag but **discards the path**. Plan 1c + T013 say "exclude the incoming feature's `OwnedPaths` from the `HasUnstaged`/`HasUntracked` rejection" — there is nothing to subtract `OwnedPaths` *from*. T013's file set lists only `TransitionReadiness.cs`/`Transition.cs`.
**Why BLOCKER:** SC-020 requires distinguishing "the only untracked path is the incoming spec" (allow one-stamp start) from "a stray untracked file is also present" (still block) — impossible without the untracked **path list**. The flagship reproduced-bug test T012 cannot pass both halves as designed.
**Resolution (plan 1c, T013 amended):** add `CommitScopeInspector.cs`/`CommitScope` to the change set — capture `UntrackedPaths` + `UnstagedTrackedPaths` lists; the readiness check computes `untrackedPaths.Except(incoming.OwnedPaths)` empty using **exact-path set membership** (not prefix); thread the incoming scope through `CommitStageTransition`. T012 already pins the stray-file-still-blocks half.

### BL-3 — T002 can silently invalidate every stamped proof via the change-set-identity hash
**Lens:** B. **Evidence:** `ChangeSetIdentity.Compute:15-26` does `OrderBy(Ordinal)` with **no dedup and no case-fold** — it trusts `ImpactChangeCollector.Collect` to return an already-deduped, `OrdinalIgnoreCase`, rename-collapsed set. Plan 0a + T002 generalize the collector to emit `ChangedFile(Path, Status, OldPath)` with a `.Files.Select(f => f.Path)` adapter, but **never state** the adapter must reproduce `Collect`'s exact set semantics (`OrdinalIgnoreCase` dedup + rename→new-path-only).
**Why BLOCKER:** any drift (a case-variant duplicate, or emitting a rename's old path) changes `ChangeSetIdentity.Compute` for unchanged inputs → every `CycleStageProof.ChangeSetId` and persisted `GateProof.ChangeSetId` reads stale, firing the staleness gate repo-wide. A silent, repo-wide proof invalidation.
**Resolution (plan 0a, T001/T002 amended):** mandate the adapter reproduce `Collect`'s exact set (`OrdinalIgnoreCase` dedup + rename-collapse-to-new-path-only); add a determinism test (extend T001) asserting `ChangeSetIdentity.Compute(builder.Files.Select(p))` equals the pre-refactor `Collect` output over a fixture containing a rename and a case-variant path.

### BL-4 — FR-039 model pinning is a no-op: the GGUF/ONNX model loads and executes WITHOUT a hash check (poisoned-model fail-open)
**Lens:** C. **Evidence:** FR-039 requires the model be "pinned + **hash-verified** like the other vendored tools" — the enforced pattern is `SentruxManifestValidator.VerifyHash` (`Sha256OfFile` vs manifest, **before** run). But plan 5b/T037 say "hash-verified via `model.version.json`" while also "keep the per-asset `RequireFile` fail-closed guard" — and `ModelLocator.RequireFile` is **`File.Exists` only**; the load path (`LLamaWeights.LoadFromFile`, `new InferenceSession(...)`) executes the file with zero integrity check. No task adds a real `Sha256`-before-load step.
**Why BLOCKER:** `HEUREX_LLM_ROOT` is an operator-writable directory outside any gate; a poisoned GGUF/ONNX is loaded into a native runtime (a memory-safety/RCE surface) unverified. This is a spec↔design fail-open — FR-039 promises verified-before-load, the design loads unverified. Per spec-is-source-of-truth, fix the **design**.
**Resolution (plan 5b, T037/T035 amended):** add a real `ModelManifestValidator` mirroring `SentruxManifestValidator.VerifyHash` — read `model.version.json`, `Sha256OfFile` the resolved GGUF/ONNX (+ tokenizer + external `*.onnx.data`), compare to the pinned digest, throw `SemanticException` (fail-closed) on mismatch/missing manifest, **before** `LoadFromFile`/`InferenceSession`; add a tampered-model-rejected test. `RequireFile` (existence-only) does NOT satisfy FR-039.

### BL-5 — the `*Planner/*Classifier/*Projector/*Detector`-in-`.Cli` invariant is asserted but does not exist (and the new assemblies aren't loaded)
**Lens:** E. **Evidence:** plan `:13` and `:168` assert ArchUnit forbids `*Planner`/`*Classifier`/`*Projector`/`*Detector` in `*.Cli`. The actual rule (`ArchitectureTests.cs:45-53`) bans only `Service/Repository/Validator/Scanner/Runner/Resolver/Engine/Provider`. 008's primary new types are exactly those four un-guarded suffixes (`RestampSafetyClassifier`, `CycleRecoveryPlanner`, `ReviewContextProjector`, `ReleaseTrainDriftDetector`, `ArchitectureRelevantSurface`). Separately, `LoadAssemblies:25-33` loads explicit assemblies — the new `Hx.Embedding.Core`/`Hx.Semantic.Core`/`Hx.Semantic.Cli` are not registered, so T035/T038's new rules would load nothing (vacuously pass).
**Why BLOCKER:** the spec's architecture clause relies on a guard that does not exist, leaving the new types' confinement unenforced; and the new boundary rules would be vacuous.
**Resolution (T035 amended; plan prose corrected):** extend `Cli_namespaces_carry_no_business_logic_types` with the four suffixes + a matching `Negative_*` entry; register the three new assemblies in `LoadAssemblies` (with a not-vacuous guard); correct plan `:13`/`:168` to describe the rule as *added by this feature*.

## HIGH (resolve in-cycle — all amended)

- **H-1 — release-train drift false positives from the working-tree union.** (Lens A) `ImpactChangeCollector.Collect` (`ImpactChangeCollector.cs:31`) **always** unions the working tree; T031's pairwise `Collect(earlier, later)` attributes in-flight/uncommitted edits to the later feature → spurious `ReleaseTrainDriftFinding` blockers at release. **Resolution (plan 3b, T031):** add a working-tree-excluding pairwise historical diff (`earlier..later`, no `git status` union) for the release-train comparison.
- **H-2 — BGE-M3 fallback port omits its real tokenizer (BlingFire) + names an unused dependency.** (Lens E) polaris-core's `BgeM3Embedder` tokenizes via `XlmRobertaTokenizer` → `BlingFireNative` (P/Invoke) + three per-RID native dlls + a CA5392 `ExcludeAssets` workaround; T036's package list omits BlingFire and names `OnnxRuntimeExtensions` (the source doesn't use it). The fallback can't tokenize as listed; SC-022 untestable. **Resolution (plan 5a, T036):** add `BlingFireNuget` + the tokenizer port + CA5392 handling + native-dll content items; fold the BlingFire natives into the pin/hash + `.sentruxignore` plan; drop/justify `OnnxRuntimeExtensions`.
- **H-3 — `Hx.Embedding` re-namespace + ArchLoader anchor wiring.** (Lens E) The port lives in namespace `Hx.Semantic.*` in polaris-core; a stray `Hx.Semantic.*` symbol left in `Hx.Embedding.Core` would pass the no-Hx-deps rule **vacuously** (same-assembly). **Resolution (T036/T035):** re-namespace the whole ported graph to `Hx.Embedding.*`; register the new assemblies in `LoadAssemblies` (shared with BL-5).
- **H-4 — `analyze` durable-report producer must exist before its `produces` binding.** (Lens A + E) FR-004 requires `/05` to **produce** `docs/reviews/{feature}-analyze-report.md`, but the analyze template (`doti-analyze.md:5`) says "Read-only … do not modify files." Binding `produces` to a stage that never writes the file → permanent Stale (`FreshnessEvaluator.cs:49-53`), reproducing the very bug FR-004 closes. **Resolution (T014):** first amend the analyze template's read-only language to carve out writing its own report (mirroring `doti-arch-review.md`'s "always write the record") and make `/05` emit it; *then* add the `produces` binding; assert the report exists post-stamp.
- **H-5 — `StaleReason` is a serialized `cycle status` field, not "internal."** (Lens B) `StageFreshnessResult` is reachable via `CycleStatusReport.Freshness` (`CycleReports.cs:20`), a `SchemaVersion` contract serialized by `doti cycle status --json`. **Resolution (plan 0b, T004):** correct the wording — it's an **additive public** field (camelCase enum string); keep `SchemaVersion`; verify no consumer asserts a closed `StageFreshnessResult` shape (or `[JsonIgnore]` it if it must not serialize).
- **H-6 — pack-time materialization can absorb a stale gitignored `.doti/templates` into the payload anchor.** (Lens B) `_HxDoti` is a filesystem glob (`Hx.Scaffold.Cli.csproj:81`); after `git rm` + gitignore, a stale local `.doti/templates/` would be swept into the staged payload whose digest is the FR-003 anchor. **Resolution (plan 4a, T033):** add `Exclude="$(RepoRoot).doti/templates/**"` to `_HxDoti`; the materializer cleans its staging target first and is the sole writer of staged `.doti/templates`.
- **H-7 — FR-021 enforcement is overstated; broaden the network ban + pin the native backends.** (Lens C) The single "no `System.Net.Http`" ArchUnit rule is presented as FR-021 enforcement but is blind to a same-assembly `System.Net.Sockets`/`WebClient` path and to the transitive LLamaSharp/ONNX/BlingFire **native** network surface. **Resolution (plan 5a, T035):** broaden the ban to the whole `System.Net.*` tree; add a task to pin+hash the per-RID native backends (the plan's risk note becomes a real task); stop presenting one namespace rule as sufficient FR-021 proof.

## MEDIUM (resolve in-cycle — amended)

- **M-1 — docs-only gate skip needs its own `GateScope` outcome.** (Lens A/C/E) `ApplyTier`/`Coverage()` are bound to the **tier config** (`GateLadder`), so a change-scope skip piggybacked on the tier `Skip` is either invisible to coverage or breaks `ValidateLadderCoverage`'s exact-match recompute. **Resolution (plan 2c, T020/T021):** model the scope skip as a distinct `GateScope`-sourced outcome+reason recorded separately from tier coverage and recomputed from the change set at validation; require the skip to be the **AND** of "affected plan = `NoTestsRequired`" **and** "ReviewContext = prose/docs-only"; `.doti/core/templates/**` is **never** docs-only (it feeds generated code).
- **M-2 — rebaseline gate: scope to the call site + make evidence forgery-resistant.** (Lens A/C) Guarding the `SentruxBaselineRunner.Save` *method* breaks first-scaffold baseline creation (`FirstSmokeRunner.cs` legitimately calls `Save` with no cycle/arch-review). And "fresh arch-review record" is forgeable (an agent-written markdown). **Resolution (plan 2e, T026):** `Authorize` guards the **rebaseline call site** (`sentrux baseline` command), not `Save` (first-creation still works); require the arch-review proof's `ChangeSetId` == current diff identity **and** a machine-checkable "functionality-driven growth" classification (not free prose); T023 covers the stale/forged-evidence refusal too.
- **M-3 — the fallback/active-engine selector is NEW logic, not a verbatim port; give it a seam.** (Lens A/D/E) polaris-core's `SemanticEngineFactory.Create(EngineId, …)` takes an explicit engine, no fallback, no active-engine recording; and both engines fail **lazily in their constructors** (`SemanticException`), so the fallback predicate must catch a locator miss **and** a load failure. **Resolution (plan 5a, T036):** an explicit `EngineSelector` (try Qwen3 construction; on `SemanticException` fall back to BGE-M3; record the active engine), unit-tested via an injected locator stub; relabel as *customized*, not ported.
- **M-4 — justify the 3-way semantic split.** (Lens D) polaris-core ships the dual-engine stack as 2 projects. **Resolution (plan 5a):** keep 3 — the split is justified by the operator's explicit "make the onnx loader a separate library so the scaffolded app can reuse it" + FR-040 (the scaffold is the named reuser of the embedders/loader). `Hx.Embedding.Core` = the reusable loader+engines; `Hx.Semantic.Core` = doti drift orchestration. Record the justification + keep the boundary clean (re-namespace per H-3).
- **M-5 — `Hx.Semantic.Cli` hosting + the FR-042 channel.** (Lens D) Every `hx doti …` subcommand is hosted in `Hx.Runner.Cli`; the `impact` precedent composes the `hx`-facing surface in the packed tool with a dev-only `*.Cli`. **Resolution (plan 5c, T039):** the FR-042 operator surface is `hx doti drift-candidates` composed into the packed tool (a thin `RunnerCommands.Doti` handler delegating to `Hx.Semantic.Core`); `Hx.Semantic.Cli` is the dev-only entry, not a second shipped tool.
- **M-6 — `SentruxOptimizationLog` new store vs a `CycleState` field.** (Lens D) **Resolution (plan 2e):** keep the separate store but record the trade-off — optimization attempts are gate-execution telemetry, not stage proofs, and folding them into the already-large `CycleState` reduces cohesion; its change-set-identity freshness stays consistent with `CycleState.BaseRef`.
- **M-7 — new diagnostics need owned `errorcodes` registry entries.** (Lens E) T041 only *verifies*; no producing task registers the refresh/recovery/rebaseline/release-train-drift codes (the freeze allows new codes, so an unregistered ad-hoc string ships silently). **Resolution (T009/T011/T026/T031):** each touches `errorcodes/registry.json` + re-render.
- **M-8 — compile-check the `ChangeSetContext`-not-in-proof invariant.** (Lens E) Plan 0a asserts it as prose; its siblings are ArchUnit-checked. **Resolution (T038/T035):** assert the gate-proof hashers exclude `ChangeSetContext`, or document the type-graph impossibility.
- **M-9 — inject the IO seams for testability.** (Lens E) `ReviewContextProjector` reads `rules.toml` via `LayerMap`; `ReleaseTrainDriftDetector` shells `git` via `ImpactChangeCollector`. **Resolution (plan 2a/3b, T017/T031):** both are constructor-injected so the projector/detector stay unit-testable (SC-011/012/013, SC-019) without a repo.
- **M-10 — additive-contract convention.** (Lens B) **Resolution (T031, T011):** new fields on serialized records (`CycleReleaseTrain`, etc.) are nullable/defaulted **trailing** params (the `CycleStageProof.PrerequisiteArtifactHashes` precedent) — additive, no `SchemaVersion` bump.
