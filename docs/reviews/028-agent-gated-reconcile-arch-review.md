# 028 — Arch-review: self-describing, agent-gated cycle reconciliation

Triage: large production-code change (the freshness/reconcile engine + the code-generated action model + the renderer migration + the `review-rebind` verb + `StageModel`) plus Doti-prose (constitution, prompts, agent-context). Nearly every code lens applies. Method: 9 clean-context lens sub-agents (design-soundness, edge-case/failure-mode, data-contract, security, blast-radius, simpler-alternative, modularity, testability, fit-with-architecture) over the spec + plan + current code → synthesis → adversarial verification. **Verdict: NOT ready for `/07-implement` — 7 BLOCKERs (all confirmed against `file:line`, none refuted), 8 HIGHs, 11 MEDIUMs.** The design needs a spec + plan revision before tasks/implement.

## BLOCKERs (block `/07-implement`)

**B1 — A bare `doti cycle stamp` bypasses the eligibility fence (the exact rubber-stamp 028 exists to close).** FR-004/D2 site the fence at "the Stamp/planner chokepoint" but realize it only as a planner *classification* enforced because `Refresh --apply-safe` filters to `SafeReinterpret`/`ReBindContentEqual` (`CycleService.Refresh.cs:39-41`). But `CycleService.Stamp` (`Stamp.cs:14-44`) is itself a public entrypoint (`doti cycle stamp --stage plan`); `EnsurePrerequisitesFresh` checks only the target's *prerequisites* (`Check.cs:67`), never the target's own `PrereqArtifactChanged`, then unconditionally rebinds `PrerequisiteArtifactHashes` (`Stamp.cs:27-28,41`). So a bare stamp after a spec edit clears `plan`'s stale with no fence, no impact assessment, no record. **Fix:** move the eligibility decision *inside* `Stamp` — distinguish a real re-author (own-artifact hash changed) from a no-op prereq-only rebind (own unchanged, only prereq hashes diverge); bare `Stamp` must **refuse** the latter on a `PrereqArtifactChanged`/non-review/`!requireChangeSetIdentity` stage and route to `review-rebind --attest no-impact`. Test: bare `Stamp` on such a stale throws.

**B2 — The verb can't read its `--target`'s own staleness from the path D2 reuses.** `Check(stageId)` evaluates the target's *prerequisite closure* (`Check.cs:67`); `CycleRecoveryPlanner.Project` projects only `report.Prerequisites` (`CycleRecoveryPlanner.cs:21`). For SC-001 (edit spec, attest on `plan`), `Check(plan)` reports clarify/specify's `OwnArtifactChanged`, never `plan`'s own `PrereqArtifactChanged`. **Fix:** `CycleService.ReviewRebind(target)` evaluates the target's *own* freshness directly via `FreshnessEvaluator.Evaluate(targetProof, requireChangeSetIdentity)` (the primitive `Status()` already uses at `Check.cs:25-31`), then runs the compound fence on that single result.

**B3 — `CycleReviewedRebindRecord` cannot live in `CycleState.Transitions`.** `Transitions` is `IReadOnlyList<CycleTransitionRecord>?` (`CycleState.cs:114`), a sealed positional record consumed as `CycleTransitionRecord` by the release-train scan. A distinct record won't compile/deserialize there and pollutes that scan. **Fix:** a separate additive nullable-trailing field `IReadOnlyList<CycleReviewedRebindRecord>? ReviewedRebinds = null` (mirroring `CompletedUnreleasedCycles`/`ReleasedCycles`), written in the same single `_store.Write` (preserves atomicity). Update spec FR-005 / plan D3.

**B4 — The plan asserts a `Hx.Doti.Core → Hx.Cycle.Core` dependency that does not exist.** `Hx.Doti.Core.csproj` references only `Contracts`, `Runner.Core`, `Version.Core` (the lone `Cycle` mention is a comment). FR-010's renderer projection (`SkillMarkdownRenderer`, `DotiRenderer`, `DotiWorkflowDescribe`) needs the renderer to reach `DotiActionModel`, forcing a **new** project edge that drags Cycle.Core's transitive closure (Runner.Core.Process, Impact.Core, GateProof, release-train) into the renderer; and `CliResults.FromActions` planned in `Hx.Cli.Kernel` would couple the domain-agnostic kernel to Cycle.Core. **Fix:** make the edge an explicit reviewed decision — (a) add `Hx.Doti.Core → Hx.Cycle.Core` (acyclic; both `order=1`) **or** (b) place the pure model in a lower leaf both already reference — and keep `CliResults` domain-agnostic by putting the descriptor→`CliNextAction` mapping in `Hx.Runner.Cli` (already refs both). Correct the plan's architecture assessment. (Couples with H8: presentation strings in the dependency-leaf engine is a layer-direction smell — reconsider placement.)

**B5 — FR-010 "zero hand-authored command info" is unsatisfiable as written.** Several `CliNextAction` sites carry payload-derived dynamic content the closed `When`-vocabulary cannot express: `ImpactCommands.NextActions` branches on `AffectedOutcome` and embeds `data.files`/`data.selectedTests` (`ImpactCommands.cs:46,86-93`); `ScaffoldCommands.Prereq` emits one action per `report.NextActions` install string (`Prereq.cs:101`); `DotiInstall` embeds runtime hook paths (`DotiInstall.cs:33,48`). None is a function of the cycle `CommandContext`, and `Impact.Cli`/`Scaffold.Cli` lack a Cycle.Core edge. **Fix:** scope FR-010/SC-010 to the **workflow-affordance class** (stage advance, recovery menu, attest verb, publish-boundary STOP, train-loop, bug-phase) that *is* a function of cycle state; carve out payload-templated next-actions as locally constructed; reword SC-010 to "zero hand-authored **workflow** command info"; scope the `new CliNextAction` guard to the affordance class via an explicit allow-list. Do **not** add an `Impact.Cli → Cycle.Core` edge.

**B6 — Deleting `skills.json nextStage` strands the 7 utility skills.** `.doti/core/skills.json` carries `nextStage` for 9 cycle stages + 7 utility skills with no `StageModel` ordinal/next edge (doti-bug/amend/converge/drift-fix/constitution/auto/upgrade; `L92/101/110/119/128/137/146`); `ResolveSkillIdentity` (`DotiWorkflowRegistry.cs:73-81`) deliberately falls back to `skill.NextStage` for exactly these. A 9-stage-only projection strands them. **Fix:** model the utility skills as `Utility`-kind descriptors with their own applicability/next, **or** keep `nextStage` for non-cycle skills and only delete it for the 9 cycle stages.

**B7 — Cascade audit-completeness (S16) is left undecided, so one attestation can silently flip multiple downstream stages Fresh.** `CascadeSafeRebindAfterStamp` (`Stamp.cs:56-86`) runs after *every* `Stamp` — including the rebind's inner content re-stamp — auto-rebinding `ReBindContentEqual` descendants with no record. **Fix:** decide — the verb suppresses the cascade for an attested rebind (and surfaces each descendant for its own attestation), **or** the cascade descendants are each recorded. Pin it in the spec; add a test.

## HIGHs

H2 — `StampedAtCommit` is nullable (`TryHeadSha`), so the FR-002 `git diff <StampedAtCommit>..HEAD` is undefined. H3 — FR-002's line-level diff has no process seam and is mis-placed on the pure `FreshnessEvaluator`; SC-004's diff-text assertion forces brittle git integration. H4 — the compound fence is assembled from three private predicates in three scopes; no single pure function to unit-test the SC-002 blocked-cases matrix git-free. H5 — `CycleService` has no injection seam (NEWs `CycleStateStore`+`StageModel` from `_repositoryRoot`); the verb + SC-007 are testable only via a real git repo. **H6 — the recovery menu is already a single-source projection from `CycleRecoveryPlanner` (`RunnerCommands.DotiCycle.cs:84/105/126`); re-routing it through a second `DotiActionProjector`/`CommandContext` reintroduces the two-evaluators-can-disagree hazard 027 eliminated.** H7 — render-from-model repoints 3 byte-stable surfaces at once with no golden baseline asserted first (the #1 risk, unmitigated as a gating task). H8 — `DotiActionModel` in `Hx.Cycle.Core` pulls presentation strings into the dependency-leaf engine (layer smell); `CommandContext` aggregating 5 read-models is a god-context.

## Resolution (applied in the revised spec + plan, 2026-06-30)

All 7 BLOCKERs + the HIGHs are resolved in the revised [spec](../specs/028-agent-gated-reconcile.md) + [plan](../plans/028-agent-gated-reconcile-plan.md):

- **B1** → FR-004 / plan D2: the compound fence (`ReviewRebindEligibility.IsAttestable`) is enforced **inside `Stamp`**; a bare `doti cycle stamp` on an attestable stale throws `Validation_CycleReviewRebindRequiresAttest`.
- **B2** → FR-003 / plan D1: `ReviewRebind` evaluates the target's own freshness via `FreshnessEvaluator.Evaluate(targetProof)`, not `Check(target)`.
- **B3** → FR-005 / plan D3: a separate additive `CycleState.ReviewedRebinds` field; one atomic write (SC-007 reframed).
- **B4 + H8** → FR-010 / plan D6 + architecture delta: explicit reviewed `Hx.Doti.Core → Hx.Cycle.Core` edge; the false dependency assertion corrected; `CliResults`/kernel kept domain-agnostic (mapping in `Hx.Runner.Cli`); deep presentation stays in `Hx.Doti.Core`.
- **B5** → FR-010 / plan D6: scoped to the **workflow-affordance class**; payload-templated next-actions (`ImpactCommands`/`Scaffold`) explicitly out-of-scope; SC-010 reworded to "workflow command info".
- **B6** → FR-010 / plan D6: the 7 utility skills become `Utility` descriptors (so deleting `skills.json nextStage` doesn't strand them).
- **B7** → FR-005 / plan D5: the auto-cascade is **suppressed** for an attested rebind; each downstream surfaces for its own attestation.
- **H1** → renderer rehome; **H2/H3** → FR-002 / D4 (diff at the CLI/recovery seam, null-safe); **H4** → the one pure `IsAttestable`; **H5** → a `CycleService` test seam; **H6** → D6 wraps the single `CycleRecoveryPlanner` projection; **H7** → a golden-baseline gating task before the render migration.

No open BLOCKER remains. Ready for `/05-tasks`.
