# Arch-Review (multi-lens) — 007 Standalone, Source-Free Hx CLI and Distribution

Date: 2026-06-26. Stage: `/06-doti-arch-review` (pre-implementation design review). Basis: spec 44 FRs / 8 SCs, plan R1–R12, tasks T001–T047. Six clean-context lens sub-agents ran in parallel, each reading the design against the **current code**. This file is the durable record; resolutions were folded back into spec/plan/tasks in the same cycle stage (commit at the `arch-review→implement` transition).

Lenses run: design-soundness, simpler-alternative/scope, security & trust boundaries, edge-cases & failure-modes, data & contract integrity, testability+fit+blast-radius. (All applicable — 007 touches production code, CLI surface, contracts, security surface, and the project graph; no lens correctly skipped.)

## Headline

The distribution core (R1–R7: drop Velopack, `PayloadRoot`, single-renderer, no-source gate, NuGet/MSIX channels, reconciliation) is test-first, well-evidenced, and architecturally sound. **Every BLOCKER is in the expanded scope (R8–R12 / FR-026–FR-044)** — it was added to a distribution spec without its own success criteria, without resolving a term collision, and without binding its new enforcement into the non-forgeable proof the way the rest of the cycle is bound.

## BLOCKERs (must resolve before `/07-implement`)

- **B1 — `--profile` is triply overloaded (term collision).** `gate run --profile` already means **lane** (`auto|advisory|normal|release`, `tools/Hx.Gate.Core/LaneResolver.cs:10-21`); `hx new --profile` means **template profile** (`ScaffoldCommandFactory.cs:44`, default `dotnet-cli`); FR-029 redefines "profile" as the **tier** (`workflow-only`/`dotnet-lib`/`dotnet-cli-heurex`) read from `.doti/integration.json`. The spec's own Acceptance line uses the OLD lane meaning. `.doti/integration.json` carries `"profile":"dotnet-cli"` — a fourth spelling. Found independently by design-soundness + testability. **Resolution:** keep `--profile`=lane on the CLI; name the new axis **tier**, resolved from `.doti/integration.json`; `GateRunner` gains a separate tier/ladder parameter; fix the Acceptance line + glossary; add a `dotnet-cli`→`dotnet-cli-heurex` migration. (spec glossary + Acceptance; plan R10; tasks T006)
- **B2 — The three curated tiers don't exist and no task defines the gate-ladder schema.** Only `.doti/profiles/dotnet-cli/profile.json` exists, with no gate-ladder field. **Resolution:** T006 sub-tasks: (a) define the `gates:{step→mode}` schema added to `profile.json`; (b) author the three tier files with concrete ladders; (c) reconcile/alias the existing `dotnet-cli` name.
- **B3 — Two descriptor files (`payload.manifest.json` beside exe vs `.doti/payload.json` in repo) with overlapping-but-divergent shapes and no binding rule.** FR-015 reconciliation compares the bundled version (in the manifest) against the repo stamp (in `payload.json`) — two unrelated numbers unless bound. Found by data-contract (BLOCKER) + design-soundness + edge-cases. **Resolution:** define two distinct records in `Hx.Tooling.Contracts`; the repo stamp's `payloadVersion` is copied verbatim from the descriptor at install; FR-015 compares those fields explicitly; add an invariant test.
- **B4 — Vendored-binary hash pin is self-referential TOFU.** `compute-tool-hashes.yml` downloads from the *same* upstream URL the fetcher uses, `sha256sum`s it, and pastes the result into the manifest — the pin is computed from the artifact it pins; no out-of-band attestation. Version drift already exists (workflow pins Sentrux v0.5.10, manifest ships v0.5.11). `ToolFetcher`'s byte-check is strong but roots in an attacker-controllable hash. **Resolution:** FR-033/T022 must verify an **upstream-published signed checksum / attestation** (Gitleaks/GitVersion publish one; the Heurex Sentrux fork must), record hash provenance in the manifest, and add a gate that fails if hash-capture version ≠ manifest version.
- **B5 — URL Trust Policy (FR-035/T034) is asserted, not designed.** A literal blocklist of `file:`/loopback/RFC1918/metadata is the classic SSRF anti-pattern — defeated by redirects, DNS-rebinding, and incomplete ranges (link-local `169.254/16`, IPv6 `::1`/`fc00::/7`/mapped, CGNAT, octal/hex encodings). Also: **no doti command ingests an operator URL in 007's deliverables yet** (simpler-alternative). **Resolution:** redesign T034 as allowlist-of-hosts + resolve-then-pin-IP + per-hop re-validation + https-only + redirects-off + content-as-data; and land it **with** the first URL-ingesting command (the bug mini-cycle's `assess` if that fetches), not free-floating.
- **B6 — Offline contradiction R3 ↔ R11.** R3 explicitly rejected "fetch binaries on first run (network dependency for the core path)"; R11/T022 moves the per-RID binaries to on-demand fetch, and `hx new`'s built-in first smoke (`FirstSmokeRunner`) runs Gitleaks/Sentrux — so the core path now needs the network. **Resolution:** decide and state — either `hx new`'s smoke degrades to advisory/Skipped when binaries are absent offline (and FR-023's `hx new` proof asserts success without the binary-dependent steps), or the **host-RID** binary is bundled (only foreign RIDs fetched). Add SC.
- **B7 — 19 expanded-scope FRs (FR-026–FR-044) have NO Success Criterion.** SC-001..008 cover only the distribution scope; the new fail-closed gates (ordered-task, profile-owns-ladder, bypass-safe) — the highest-risk new enforcement — are provable only by manual review, which the constitution forbids reporting as proof. **Resolution:** add SC-009..SC-018 (below).

## HIGH

- **H1 — `GateProof` doesn't bind the tier/ladder.** Validation recomputes the affected-test plan but not the gate ladder; a tier that drops `sentrux-check`/`architecture-test` still mints a valid, fresh, passing proof — a silent downgrade primitive (security + testability). **Fix:** add an active-tier + ladder-coverage hash to `GateProof`; recompute it in `GateProofValidator` exactly like the affected-test plan; refuse a proof minted under a narrower-than-declared ladder. (Folds in the security "edit-the-profile bypass" finding.)
- **H2 — Ordered-task enforcement (T003) is greenfield mis-scoped as "extend."** `DotiTaskCompletion` has no phase/predecessor/`[P]` concept; "prior phase gate green" has no substrate and risks circularity (the gate already includes the task-completion step). **Fix:** add a task (before T003) defining a machine-readable phase/predecessor schema in the tasks template; split T003's two halves (the predecessor-prefix half reuses the parser; redefine "prior phase green" as "prior phase's tasks all checked + hash-valid" to break the cycle); key ordering on `TaskId`, not `LineNumber`.
- **H3 — `hx new` source-vendor gap.** `ScaffoldNewRunner`→`SourceVendor.Vendor` copies 12 `tools/Hx.*` **source** projects into every generated repo; a source-free installed `hx` has none. T021 says "re-point off the source tree" but never says what `SourceVendor` resolves to. **Fix:** decide explicitly what a generated repo's workflow tooling becomes (compiled binaries? a `dotnet tool` ref? the pack carries it) and state it in T021/FR-032.
- **H4 — `hx.config.json` missing from the global-tool payload.** Every operational command is wrapped in `WithRequiredConfiguration`; T023's payload list omits `hx.config.json` (T025's MSIX list includes it) → SC-001 fails closed. **Fix:** add it to T023 (and confirm read-only `version`/`prereq` tolerate it per FR-021).
- **H5 — FR-044 patch→minor flip desyncs from the GitVersion `+semver:` trailer.** Two independent defaulters: `LocalReleaseService.NormalizeReleaseIntent` (T041 flips to minor) and the cycle transition that writes `+semver:` only on an explicit `--release-intent`. A blank intent → no trailer → GitVersion calculates patch → `ValidateIntent` throws "Release intent mismatch" by default on the first post-007 feature release. **Fix:** make the cycle transition write `+semver: minor` by default for a feature cycle (no signal/patch for a bug-fix-only cycle); add a test.
- **H6 — FR-003 fail-closed only on *absent* descriptor.** A present-but-corrupt / wrong-`schemaVersion` `payload.manifest.json` throws a raw `JsonException` instead of a structured `Integrity` `CliResult` (the existing `StageModel.Load` already shows the fail-closed pattern). **Fix:** FR-003 → "absent, unparseable, or unsupported schemaVersion"; add cases to T009.
- **H7 — Renderer hardening test passes by omission.** `CliHelpRequestParser` doesn't handle `--version` or unknown-*subcommand*; T013 enumerates only the *current* known commands and unknown-*option*, so a later command (e.g. `doti bug assess`) silently regains the SCL fall-through. **Fix:** make T013 structurally fail-closed (forbid any reachable `HelpAction`/default error action on the parsed tree; assert every `RootCommand` routes through `CliApp.Harden`); add unknown-subcommand + `--version` to T013/SC-006.
- **H8 — Brownfield first-install undefined.** FR-015 has only equal/older/newer — all presume a recorded version. A repo with no `.doti/payload.json` (every pre-007 repo, the headline FR-030 case) hits an undefined branch → NRE or blind overwrite. **Fix:** add an "absent recorded version → fresh install through the conflict-aware path, never blind-overwrite" branch + T029 case.
- **H9 — `hx doti install` is not idempotent/resumable.** `DotiInstaller.Install` writes files then `WriteBaseline` last, non-atomically; a crash leaves files with a stale/absent baseline → next reconciliation clobbers operator edits. **Fix:** stage-and-swap or write baseline/stamp transactionally; add a resumability test.
- **H10 — `release.yml` OIDC publish blast radius.** Triggers on any `v*` tag push; T024 adds `id-token: write` + Trusted Publishing → anyone who can push a tag publishes to public NuGet; actions pinned by floating major; the fallback stored API key is the posture OIDC exists to avoid. **Fix:** environment-gate the publish job with a required reviewer (or protected tag pattern); pin actions to full SHAs; drop/isolate the fallback key; `id-token: write` only on the publish job.
- **H11 — `HX_PAYLOAD_ROOT` override = unauthenticated payload substitution.** The override takes precedence over the beside-exe anchor and redirects *everything* trusted (templates, prereq policy, the tool manifests whose hashes gate fetched binaries). **Fix:** `PayloadRoot.Resolve()` integrity-checks the descriptor regardless of source; emit a loud diagnostic when the override is active. (Collapses with B3/B4 — the manifest is the un-anchored trust root in all three.)
- **H12 — `hx doti install` path-traversal.** Managed writes + `.new` sidecars aren't validated to stay inside `--repo`; a `..`/absolute/symlink managed path escapes (the repo already has `RepositoryPathResolver.ResolveInside`). **Fix:** route every managed write through `ResolveInside`, fail closed on escape; treat the payload manifest's path list as untrusted; add a traversal test.
- **H13 — `Living-Spec` relaxation can let stale plan/tasks pass as fresh.** Freshness is per-artifact-hash only; the `plan`/`tasks` proofs don't bind the upstream `spec.md` hash, so editing the spec and skipping re-analyze leaves them "fresh" (edge-cases + testability). **Fix:** a doc stage's proof must also bind the content hash of each prerequisite artifact at stamp time; add an SC + a regression test that the relaxed path still blocks a genuinely stale dependent.
- **H14 — Impl tasks with no preceding test.** The doc claims "a test precedes the implementation it covers," but T002, T003, and all of Phase 6 (T033–T041, incl. the URL-trust security control T034) have none. **Fix:** add preceding test tasks for T002/T003/T034/T041 (at least), or correct the global claim to scope it to the distribution + reconciliation phases.

## Notable MEDIUM / LOW (resolve at implement; recorded so they aren't lost)

- New error codes unnamed/unregistered: `Validation_TaskOutOfOrder`, `Validation_ProfileGateMissingConfig`, `Validation_UrlBlocked`, plus "bug-cycle/converge codes" are not in T005's scope and will be invented ad-hoc against the **append-only frozen** registry — and the spec's PascalCase names are the *generated* form, not the registry `category`/`suffix` shape. **Fix:** enumerate every new code (in registry shape) in T005/per-producer before its task.
- `Validation_DotiRepoPayloadAhead` exit-class is inconsistent with its two `Integrity_*` siblings in the same feature (precedent cuts both ways: `tool-store-version-too-new` is Validation; the managed-asset refusals are Integrity) — pin it with a one-line rationale in T005.
- `LocalReleaseResult` Velopack→channel field churn silently breaks `release.identity.json` and orphans the frozen `ITG0008 integrity.velopack-artifact-missing`; no schemaVersion strategy (`JsonContractDefaults.SchemaVersion` is one shared constant). **Fix:** make the rename additive/channel-neutral; decide ITG0008's fate; state the schema decision.
- No-source gate should be an **allowlist of expected entries** scanning the *recursively staged* artifact (and *inside* the template `.nupkg`), not a denylist of markers — `artifacts/**/msix-layout/src/Hx.*` already exists on disk from the old `git archive HEAD`. (T019/T020)
- Profile-composition algebra (T008 overrides→profile→core / prepend/append/wrap) is premature generality for 3 fixed tiers — reduce FR-029/030 to a `gates:{step→mode}` map; defer composition until a real 4th profile needs it.
- Concurrent vendored-binary fetch has no lock (temp-then-rename); offline fetch surfaces a raw `HttpRequestException` not a structured `CliResult`; equal-version repair must run the same hash-preservation as forward; managed-file deleted/renamed-upstream states are unhandled; stale Velopack error string at `ScaffoldCommands.DotiInstall.cs:35`; the `velopack` entry in `ToolFetcher.ManifestRelativePaths` must be removed by T017 or `FetchAll` fails for all tools; the `.doti` re-render (T043) should land with the Velopack removal (T016/17) so the skill-drift gate doesn't trip mid-cycle; collapse `ScaffoldRoot`'s dead `AppContext.BaseDirectory` fallback so only `PayloadRoot` is executable-anchored; unsigned local/CI MSIX must be Store-submission-input-only and never release-attached (mirror the no-source gate).

## Scope / split (operator-decided — not applied)

Two lenses recommend splitting the 47-task program into **Feature A (distribution core, ships first), B (enforcement substrate), C (Spec Kit absorptions)**, because the distribution core has *zero* code-dependency on the substrate (the stated "substrate-first" rationale is backwards) and T003 would gate the urgent hotfix behind the speculative Phase 0. The operator has deliberately chosen to **capture the whole scope in one spec (007) and split later, preserving phase order** — which the lens explicitly endorses as viable ("the capture goal is satisfied by the document; it does not require one release train; split into cycles at implement time"). **Decision: keep 007 as the single capture; revisit the split at implement-time.** The dependency-rationale wording in the tasks doc is corrected so a later split is clean.

With them resolved, no BLOCKER remains open in an applicable lens → `/07-implement` is unblocked. The HIGH/MEDIUM items are recorded here and in the revised tasks so implementation carries them.

## Second round — independent Codex panel (2026-06-26)

A second six-lens review (run via Codex) found 5 further BLOCKERs + 6 non-blockers. **All were judged valid** (several were items the first round *raised* but left as "decide", and one the first round got *wrong*). Verified against code before accepting; all folded in:

- **C1 — Generated repos lose their workflow path (the big one; first round got it wrong).** `SourceVendor.Vendor` copies the **12-project runner+impact closure** into every generated repo because the repo's whole doti workflow runs as `dotnet run --project tools/Hx.Runner.Cli -- gate run`/etc. (generated `profile.json` ≈ 34 such commands; `FirstSmokeRunner.BuildVendoredTooling`). The installed `hx` (`Hx.Scaffold.Cli`) does **not** expose that surface. So the first round's "remove `SourceVendor`, use `hx`" would strip the repo's workflow. **Fix: new FR-045** — wire the runner+impact command trees into installed `hx`; rewrite generated `profile.json`/agent-context/smoke to `hx <command>`; remove `SourceVendor` only after. (FR-045, SC-019, T015, T021, T027)
- **C2 — Descriptor had no concrete trust root.** "Integrity-verified" was metadata-only → a malicious payload could carry a self-consistent descriptor. **Fix:** executable-embedded expected payload digest + per-file hashes, verified on every path incl. `HX_PAYLOAD_ROOT`. (FR-003, R2, T010)
- **C3 — SC-004 banned the template pack's own content.** A blanket `.csproj`/`src/` ban false-positives on the required template pack. **Fix:** SC-004 rewritten to ban the tool's own build tree / full source archive, allow expected template-pack content (FR-005-consistent). (SC-004)
- **C4 — FR-031 still MUST-required composition the plan/tasks deferred.** **Fix:** narrowed FR-031 to the concrete `gates:{step→mode}` map; composition deferred beyond 007.
- **C5 — Contract version strategy undecided.** **Fix (decided):** additive/channel-neutral `LocalReleaseResult` rename, schema stays v1, `ITG0008` orphaned-but-frozen, compat test; two payload records carry their own `schemaVersion`. (T004/T028)
- Non-blockers folded: offline degrade restricted to genuine network conditions (hash/provenance/malformed/extract fail closed); Store submission hardened like NuGet (environment/reviewer, SHA-pin, isolated creds); **package strategy → framework-dependent** (RID-specific self-contained is moot once binaries are fetched); `Integrity_ProfileGateMissingConfig` aligned across SC-011/T005/T007; template Velopack-free asserted by the T027 smoke (catches the Phase-7 template fix early); T018 authored test-first.

Net: 0 BLOCKERs open after both rounds; `/07-implement` unblocked.
