## Execution order — phases are SEQUENTIAL (mandatory)

**Implement strictly top-to-bottom, by phase and by task number. Work only the lowest-numbered incomplete task.** A phase is "done" only when its tasks are checked **and** `gate run --profile normal` is green; the next phase starts only then (T029 is the explicit final gate; run it at each phase boundary too). **Test-first:** for code areas the `[test]` task precedes the implementation it covers; doti-prose (templates/skills/constitution) tasks carry their test inline (a render/parity/cycle-state assertion).

**Load-bearing sequencing notes (from the plan):**
- The §1/§2 **template** (T002) and **this repo's constitution** (T003) are the source of truth the skill, the emit command, and the injection all reference — they come first.
- **WI1–4 (constitution) and WI5 (.NET finder) are independent** — Phase 5 can be worked in parallel with Phases 1–4 by a separate context, but the gate at each phase boundary is shared.
- **R5 (verified bug):** the payload glob ships *this* repo's `.doti/memory/constitution.md` and the installer reconciles `memory/`, so a generated repo currently inherits speckit-doti's constitution. T018 excludes it from the payload and initializes from the template — land T017/T018 paired.
- The constitution is a **project artifact, not a per-cycle stamp** (FR-003): it is NOT added to `workflow.yml`; the `doti-constitution` skill is unnumbered.
