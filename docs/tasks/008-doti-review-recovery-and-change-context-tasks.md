# Tasks: Doti Review Recovery and Change Context

Plan: `docs/plans/008-doti-review-recovery-and-change-context-plan.md`. Spec: `docs/specs/008-doti-review-recovery-and-change-context.md`. Tasks are organized into **mandatory sequential phases (Phase 0 → Phase 5)** and numbered **T001 → T041 in execution order**. Each `FR-###`/`SC-###` maps to ≥1 task. **Priority mode = workflow/tooling: safety-first** — fail-closed proof integrity (P1) before context/scoping (P2), completeness (P3), templates (P4), and the advisory semantic finder (P5, last).

## Execution order — phases are SEQUENTIAL (mandatory)

**Implement strictly top-to-bottom, by phase and by task number. Work only the lowest-numbered incomplete task.** A phase is "done" only when its tasks are checked **and** `gate run --profile normal` is green; the next phase starts only then (T041 is the explicit final gate; run it at each phase boundary too). **Test-first:** for code areas the `[test]` task precedes the implementation it covers; doti-prose (templates/skills) and `workflow.yml` data tasks carry their test inline (a render/parity/cycle-state assertion).

**Load-bearing sequencing notes (from the plan):**
- Phase 0 types (`ChangeSetContext`, `StaleReason`, `FeatureArtifactScope`) are shared foundations for Phases 1–3 — they come first.
- **T014 must land paired with T009:** adding `produces:` to the empty-hash review stages instantly stales them (`FreshnessEvaluator` empty-`ArtifactHashes`); the empty→bound transition is precisely a `SafeReinterpret` that T009's `--apply-safe` resolves cleanly.
- FR-033/034/035 (multi-spec release train) are **already implemented + verified** (starting 008 finalized 007 as completed-unreleased) — T027 is a regression test that pins them, not a build.
- The new-feature-start scoping bug (FR-038) and the `analyze` stale-loop (FR-004) were **reproduced this session** — T012/T013 and T014 close them.
