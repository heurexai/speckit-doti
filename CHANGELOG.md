## [Unreleased]

- **Added** `039-git-transaction-envelope` (feature) — engine-owned git-state integrity across the cycle *and* the release, so a coded doti git operation either completes fully or is automatically reverted, never leaving orphans, empty commits, or leftover tags/dirs, and never relying on the agent to clean up. **WI1 (transition integrity):** the coded stage transition now stages the leaving stage's declared `produces` artifact itself (scoped, never `git add -A`) and fails closed only on a genuine orphan — no more empty transition commits leaving an authored spec untracked (the doc-dance bug that surfaced repeatedly driving this very cycle). **WI2 (release transaction envelope):** `hx release` runs inside a compensation ledger (`ReleaseTransaction` in `Hx.Runner.Core/Git/`); on *any* stage failure it automatically reverts its own durable side effects — the created tag (`git tag -d`), the release-root/`<version>` dir (restored-to-baseline, cross-volume-safe), and the `.doti/cycle-state.json` bytes — and returns a fail-closed `RollbackReport` (no throw), so a failed release leaves the repo byte-identical to before, no leftover tag/dir poisons the retry (the ergon-v0.1.2 blockers), and an agent adding debug files to a half-failed release is impossible because the engine reverts immediately. A `[ThreadStatic]` fault-injection seam + a named `ReleaseStage` pipeline make the byte-identical revert deterministically testable without dotnet. **WI4 (lifecycle finalize):** a new `hx doti cycle finalize-release` recovers a cycle wedged at the release stage — the dev→main→CI publish path never calls `MarkReleaseTrainReleased`, so `ReleasedCycles` stays empty and the next feature can't start (the 031 wedge) — moving the released feature into `ReleasedCycles` without re-validating the shipped train, surfaced by `doti cycle status` so no cycle-state is a dead-end. The design was reshaped by a 4-lens architecture review that caught (pre-code) a compile-breaking module-placement cycle, an FR that would have fail-closed a legitimate `clarify→plan`, and an *unsafe* whole-tree snapshot that would have clobbered concurrent operator edits and dropped exec-bits — the corrected design is a small compensation ledger over the operation's OWN side effects, not a repo snapshot. New error code `Integrity_ReleaseReverted` (ITG0017); 11 new tests; recorded under `docs/specs/039-git-transaction-envelope.md`.
- **Fixed** `038-bug-only-release-cycle-check` (bug mini-cycle): `hx doti cycle check --stage release` was **not bug-only-aware** — on a repo with no `.doti/cycle-state.json` (a bug-only release, e.g. ergon shipping only `/doti-bug` mini-cycles) it marked **all eight** feature stages (`specify`…`drift-review`) *"missing"* and failed, a **false blocker** that misled a release agent into proposing it *fabricate* a feature cycle — the exact anti-pattern `033`/`034`/`037` exist to prevent. Root cause: the `033` bug-only tolerance (`BuildReleaseTrain` admits a null cycle-state and consults the injected bug members) was **never extended to the `Check()` chokepoint** — its release-train block is guarded by `state is not null`, so a bug-only repo never reached it and fell through to the transitive-prerequisite loop, where `state is null` marks every prerequisite missing (the same "evaluation widened for bug-only, chokepoint not" gap class as `033`/`034`). Fixed: a release-scoped null-state branch in `CycleService.Check` (`BugOnlyReleaseCheck`) that delegates readiness **entirely** to the bug-aware release train — the same evaluation `GetReleaseTrain`/`MarkReleaseTrainReleased` already trust — instead of feature prerequisites a bug-only repo neither has nor should fabricate: a valid train (≥1 test-passed, unreleased bug) passes with a stand-in `release-train:ready` result; an empty/invalid one **fails closed** on the train's own blocker, so a genuinely-empty repo is still rejected. Scoped to `target.Id == "release"` only, so feature-stage checks are unchanged. Proven **end-to-end on the real ergon repo** (bugs `004`+`005`): the check now returns `passed:true` with both bugs `included`, where pre-fix it failed *"Missing/not stamped: specify clarify plan arch-review tasks analyze implement drift-review"*. **Plus a coupled write-path hardening an adversarial (Opus, 3-lens) verification of the fix surfaced** — one the 149+260 green unit tests missed: `CycleService.Stamp` consumes the *same* `Check` chokepoint via `EnsurePrerequisitesFresh`, so making the release READ bug-only-aware would have let `doti cycle stamp --stage release --feature <x>` on a bug-only repo *pass* and `_store.Write` a **fabricated feature `cycle-state.json`** — the precise anti-pattern 038 closes (and the `/09` skill instructs exactly that stamp). Fixed with a `RefuseBugOnlyReleaseStamp` guard: a `release` stamp on a null-state (bug-only) repo fails closed *before the first git touch* — a bug-only release is never stamped (033: the release path never calls `Stamp`; it self-maintains shipped status by tag-reachability), while a legitimate feature-cycle release stamp (non-null existing state) is untouched. 4 new `BugOnlyReleasePathTests` (valid-train passes without feature stamps; no-releasable-bug still fails closed; feature-stage check is not short-circuited — release-scoped; **and the release-stamp guard fabricates no cycle-state**); full `Hx.Cycle.Tests` 150/150, `Hx.Runner.Tests` 260/260. Recorded under `.doti/bugs/038-bug-only-release-cycle-check/`.
- **Fixed** `037-release-non-hx-product` (bug mini-cycle): `hx release` could not release a **non-hx** scaffolded product (e.g. ergon) — three coupled bugs that never surfaced because speckit-doti only ever releases *itself* (hx), so the vendored-hx-releases-the-target's-product path (the `.doti/release.json` feature) had no coverage. - **Fixed** `036-reconcile-dir-candidate-commit` (bug mini-cycle): a regression from `035`'s pathspec-scoped sanctioned commit. `DotiReconcileCommit.TouchedPaths` records **directory** candidates (`.doti/core`, `.doti/profiles`, `tools/sentrux`, …) for whole managed subtrees; `git add -- .doti/core` stages the files under it individually, but `SanctionedGitCommit`'s commit pathspec used a file-EXACT intersection, so `.doti/core/skills.json` + a materialized template were **staged but not committed** — a consumer `hx doti update` left managed source files staged-uncommitted (found **verifying** an ergon 0.18.3 update — the pre-035 whole-index commit had masked it). A regression test reproduces the exact `.doti/core` case + re-proves the out-of-tree operator-file protection. Recorded under `.doti/bugs/036-reconcile-dir-candidate-commit/`.

(E) `doti update-all` counted a repo whose commit failed as **both** updated and failed — the buckets are now one mutually-exclusive disposition. (F) the transient-lock retry now wraps `git add`, not only `git commit`. **Release-train / gate integrity:** (G, HIGH) the `release-documentation` gate matched a member slug with a substring `Contains`, so `032-x` was satisfied by an unrelated `1032-x` — now a bounded-token match. (H) the bug-test "no evidence-free pass" rule was enforced only on write, not re-checked by `ReleaseReadyBugMember` at the release trust boundary (a hand-edited `test.json` bypassed it) — now re-checked. (I) one malformed bug-record JSON threw an uncaught `INT0001` out of every release-train command — a corrupt record is now fault-isolated as not-a-member. (J) shipped-ness keyed on the highest-semver tag regardless of ancestry, and on the record dir's last commit — now constrained to a `--merged HEAD` tag, and a working-tree-dirty bug record is treated as re-openable. The load-bearing part: **the regression tests that were missing** — a PRE-STAGED operator file (the prior tests used working-tree-only edits that never exercised the sweep), a gitignored `.doti/templates/`, an untracked orphan, the bounded-slug match, the evidence-free pass, a truncated record, an unmerged-branch tag. - **Fixed** `034-bug-only-release-doc-commit` (bug mini-cycle): a bug-only repo (a confirmed `/doti-bug` mini-cycle with no numbered feature cycle, hence no `.doti/cycle-state.json`) had **no coded sanctioned path** to commit the release-documentation fix its own release gate demands. `033` widened `gate run --profile release` / `CycleService.GetReleaseTrain` to *run* on a null cycle-state, so a bug-only release now reaches the `release-documentation` step — which requires the bug's slug in `README.md`/`CHANGELOG.md` — but nothing coded could commit that staged fix: the insurance pre-commit hook blocks a bare commit, `doti cycle commit` **never existed** (a stale-spec mirage; the `doti cycle` tree is `stamp`/`status`/`check`/`refresh-plan`/`refresh`/`review-rebind`), `hx doti bug` assess/fix/test write only JSON, and `hx release` makes no commit — so `033` widened the gate's *evaluation* to admit a bug-only repo without widening the matching *commit* path. - **Fixed** `032-doti-update-worktree-leak` (bug mini-cycle): two coupled defects in `hx doti update`. (1) **Worktree-leak / commit-race / silent swallow** — the self-commit (`DotiReconcileCommit`) ran while the reconcile's own temp worktree was still registered (`DotiWorktreeUpdate` disposed it in `finally`, *after* the commit), so on a worktree-cluttered repo the commit lost a shared-`.git` lock race, `git commit` failed, and the failure was reported as `DotiCommitStatus.Failed` but **swallowed under `ok:true`/exit-0** (changes left staged). Fixed: the worktree is now fully disposed *before* the commit (extracted `RunWorktreeScoped`); a `Commit.Status == Failed` routes to `CliResults.Fail` (`ok:false`/non-zero/`Integrity`) for both `update` and `update-all`; a new **ownership-aware** `GitWorktree.PruneLeakedTemps` reclaims leaked `hx-doti-worktree-*` temps at run start + in `Dispose` (`git worktree remove --force`, else `rm` the corrupted husk, then `git worktree prune`, with a 10-min age floor so a **concurrent** run's live worktree is never killed — a real bug caught during the fix); and the commit retries on a transient `index.lock`/`cannot lock ref` signature before reporting `Failed`. (2) **Vendored-tool reconcile gap** — `tools/{sentrux,gitleaks,gitversion}` are scanned into `managed-assets.json` but the reconcile + parity loops iterate a hard-coded `.doti/{sub}` list that excludes `tools/`, so a doti release that bumps the pinned Sentrux (v0.5.11→v0.5.12) left consumer repos on the old tool with a silently-broken gate (`"Could not read the Sentrux gate baseline signal"`). Fixed: a parallel `tools/{sub}` reconcile pass in `DotiInstaller` (reusing the existing preserve/`.new` machinery; `bin/` excluded so the gitignored exe is never touched) + a `CheckToolFiles` parity check (kind `vendored-tool`) so `payload check` flags a stale manifest/grammar + a `DotiUpdater` advisory to run `hx tools fetch`; the exe stays offline-fetched (`SentruxManifestValidator.Verify` already fail-closes on a hash mismatch → a clear stale-binary error). 16 new deterministic tests; recorded under `.doti/bugs/032-doti-update-worktree-leak/`.

- **Added** `031-doti-update-self-contained`: `hx doti install` / `update` / `update-all` are now **self-contained** — one command from any directory reconciles the target repo's managed Doti assets, prunes orphans, and commits, with no manual workarounds. (1) **Bundled-source default** — a new `BundledPayloadResolver` resolves the source to the **running tool's bundled payload** beside the executable (the global-tool's payload root), falling back to the working-directory `FindDotiSource` walk only for a genuine dev/in-repo source, and **failing closed** (`Validation_DotiPayloadSourceUnresolved`) when neither resolves — so the commands no longer require a `cd` into the tool's bundled-payload dir, and the false `already-current` is resolved by construction (the bundled `payload.manifest.json` stamps the real version → honest before≠after). (2) **Complete reconcile** — `DotiInstaller.PruneOneOrphanDir`'s no-baseline branch now **prunes** a `*doti-*` rendered-skill orphan the render no longer targets (rendered skills are not operator content; hand-edits are drift) instead of preserving it, so a cross-version skill rename (e.g. the 026 reorder) leaves no husk and `doti payload check` passes; operator-**policy** assets (`skills.json` under `managed-replace-preserve-live-config`, the constitution) stay preserved. `StageSidecar` skips the `.new` when the bundled content is byte-identical to the operator's file (no spurious stray), and the staged `.new` sidecars are surfaced as a distinct **merge-pending** list. (3) **Self-owned commit** — a new `DotiReconcileCommit` makes, after a successful reconcile in a Git work tree, **one sanctioned commit of exactly the touched managed-asset paths** (`Installed ∪ Removed ∪ Rendered`, minus `.new` + gitignored runtime state — never `git add -A`), setting `DOTI_SANCTIONED_COMMIT=1` on the child `git commit` so the insurance hook permits it, with an auto message naming the before→after move + pruned orphans; idempotent (no staged change → no commit), non-git skips it, and a new `--no-commit` flag opts out. The resolved-source origin, pruned paths, `.new` merge-pending list, and commit outcome (sha / skip reason) are reported in the `--json` envelope + human summary. (4) **Bug-cycle RCA discipline** (`/doti-bug` skill + `speckit.bug.{assess,fix,test}` templates): assess now requires a proper **reproduce → root-cause → validate** RCA; fix must target the **root cause** and explicitly **forbids** a symptom patch/bandaid/workaround; the agent **does** the root fix and must **not** present a bandaid-vs-root choice or ask which fix to apply (surfacing only a genuine blocker). (5) **Doc-stamp scope consistency** (resolves #42): `CycleService.ValidateTransitionReadiness` excludes the active feature's owned produces docs (`FeatureArtifactScope.OwnedPaths`) from its untracked/unstaged scope guards exactly as `Check()` does, so authoring a stage's produces doc ahead of stamping the transition into it no longer trips the set-aside/stage/restore dance — a genuinely foreign untracked path still blocks. Logic lives in `Hx.Doti.Core`/`Hx.Cycle.Core`; the CLI stays parse→delegate→render; contract changes are additive trailing-optional; no new dependency edge.

Both are 1:1 over one schema, **validate fail-closed before any file is generated** (an invalid value names its field and leaves no partial repo, SC-006), and persist the operator's intent — repo-portable fields only, never a machine-local absolute path — to a tracked `.doti/setup.json`. The config projects into `GitVersion.yml` (`next-version`), the CLI `.csproj` (`<Description>`/`<RepositoryUrl>`/`<PackageLicenseExpression>`/`<Authors>`, XML-encoded against MSBuild injection), and constitution §2 (write-once-when-placeholder). **`hx doti config show [--json]`** renders the effective config grouped by what each setting drives, marking every key **default vs custom** (the model carries `value`/`source`/`default` per key); with no `.doti/setup.json` it shows the all-default view. - **Fixed** `030-bug-release-bridge`: a single completed cycle can now anchor its own release. `CycleService.BuildReleaseTrain` previously counted only completed-unreleased feature cycles plus the active cycle *at* `release`, so a feature parked at `drift-review` (or a bug-fix-only repo) produced an empty, invalid train and `hx release` / `gate run --profile release` refused to proceed. - **Added** `028-agent-gated-reconcile`: self-describing, agent-gated cycle reconciliation + a code-generated workflow action model. (1) **Agent-gated reviewed-no-impact rebind** — 027 stopped the engine from silently rebinding a content change; this stops the *agent* from rubber-stamping the forced re-run. When an upstream artifact's content changes, a downstream non-review, non-code-bound stage is now a `RestampSafety.ReviewedNoImpact`: the engine surfaces the changed prerequisite paths + the line-level upstream `git diff` (FR-002 — lazily, at the CLI/recovery seam, not the pure evaluator), and the agent must make an explicit IMPACT DECISION — re-author if affected, or `hx doti cycle review-rebind --target <stage> --attest no-impact [--reason]` if not. The eligibility fence (`ReviewRebindEligibility.IsAttestable` = `PrereqArtifactChanged` AND non-review AND `!requireChangeSetIdentity`) is enforced **inside `Stamp`**, so a bare `hx doti cycle stamp` of an attestable stale **refuses** (`Validation_CycleReviewRebindRequiresAttest`) — closing the agent-rubber-stamp hole 027 left open. The verdict is recorded immutably (dependent, changed upstream, before→after hashes, reason) in a new additive `CycleState.ReviewedRebinds` field (one atomic write); it **decays** automatically when the reviewed upstream later moves again; the auto-cascade is **suppressed** so one attestation never silently flips multiple stages Fresh; `OwnArtifactChanged`/`ChangeSetDiffers`/review-kind/change-set-bound stales can never be attested away. (2) **Code-generated workflow action model** — the agent's valid *workflow* next-action commands (stage advance, the recovery menu, the attest verb, the publish-boundary STOP, the train-loop, the bug-phase, the 7 utility skills) are now GENERATED from one `DotiActionModel` registry of `CommandDescriptor`s with **declarative** applicability (a closed named-condition vocabulary, not opaque predicates) over a `CommandContext`, projected onto `CliResult.NextActions`, the recovery menu (**wrapping** the single `CycleRecoveryPlanner` projection — never a second evaluator), the rendered skill refs, and the agent-context `{commandAvailability}` placeholder. It **subsumes `DotiWorkflowRegistry`** (deleted; presentation rehomed in `Hx.Doti.Core` over a new acyclic `Doti.Core → Cycle.Core` edge) and deletes `skills.json nextStage` (utility skills become `Utility` descriptors); payload-derived next-actions (`ImpactCommands`/`Scaffold`) stay locally built. `next:` is declared on all 9 stages + bound in `StageModel` (the `CheckHelpers` DFS de-duplicated onto `StageModel.TransitivePrereqStages`). (3) The §1 **`Self-describing automation`** constitution invariant — the program does the bookkeeping + surfaces the evidence AND the next-action; the agent only judges. Designed through the reordered cycle: a multi-lens `/04-arch-review` caught **7 BLOCKERs** (incl. the bare-`Stamp` fence bypass) + resolved them *before* implement. Full suite green; render parity clean.
- **Fixed** `027-stamp-reconcile-orphan-prune`: codified stamp reconciliation + the `hx doti update` orphan bug. (1) **Freshness cascade** — re-running the one genuinely-changed stage now AUTO-CASCADES a content-equal rebind to its content-unchanged dependents, ending the stage-by-stage hand-re-stamp treadmill. The vendored tool manifests gained linux-x64 + osx-arm64 assets (the Heurex sentrux fork ships every platform), and `ToolFetcher` now extracts `.tar.gz`, not just `.zip`.

## [0.1.0] - 2026-06-21

First tagged release — the toolkit is published as a downloadable, standalone win-x64 installer.

### Added
- **Standalone installer** — a self-contained `hx.exe` plus the bundled scaffold payload (template, doti workflow, source projects, and the vendored tool binaries), published as a GitHub Release asset. `hx new` scaffolds a project from any directory (the payload is resolved relative to the executable).
- **Shared tool store** — the vendored binaries (Gitleaks, Sentrux, GitVersion) install once into a per-user data folder with an environment override; generated solutions resolve them store-first (in-repo fallback) instead of carrying a ~127 MB per-solution copy.
- **Thin-CLI architecture enforcement** — a *Channel Independence (Thin Adapter)* constitution principle, plan/arch-review channel-boundary checks, and two new ArchUnit families (`cliSurfaceConfinement`, `cliDelegation`) shipped in the generated template (nine families total) and dog-fooded on the toolkit's own `Hx.*.Cli` projects.
- `tools fetch` — deterministic, hash-verified provisioning of the vendored tool binaries from their pinned manifests (fail-closed on mismatch).
- Agent-first CLI self-description (structured `<PREFIX><NNNN>` error codes, `describe`, the `CliResult` envelope) encoded into the doti workflow.
